# Enterprise Code Design Skill

`enterprise-code-design` 是一个用于企业级软件交付的 Codex Skill，简称 `ECD`。

它的目标不是让 AI 更快地“直接开改”，而是让 AI 在复杂任务中先理解项目、识别风险、设计方案、控制影响范围、主动验证，并把重要过程沉淀成中文文档。

> 说明：本 README 面向人类使用者，帮助理解 ECD 的能力边界和使用方式。真正的 Skill 触发规则与执行规则以 `SKILL.md`、`references/rules/reuse-mode.md`、`references/rules/review-mode.md` 为准。

## 核心理念

ECD 默认遵循以下工程原则：

- 先查阅项目和现有实现，再做设计和修改。
- 先确认不确定点，再处理高风险或中高复杂度任务。
- 优先复用现有架构、接口、模块、组件、工具和规范。
- 以最小影响范围完成任务，不做无关重构、格式化或顺手清理。
- 对公共契约、状态、数据、权限、安全、性能、部署和核心链路保持谨慎。
- 需要验证的地方主动运行 lint、测试、构建、脚本、浏览器检查或等价验证。
- 所有面向用户的说明、方案、交付总结、评审记录和新增代码注释默认使用中文。

## 适用场景

适合使用 ECD 的任务包括：

- 复杂业务功能开发。
- 前端、后端、移动端、桌面端、数据/ML、基础设施、自动化脚本、文档系统等非平凡改造。
- 多模块、多文件、跨层级的代码调整。
- API、DTO、CLI 参数、事件、文件格式、数据库 schema、迁移脚本、部署配置等公共契约变更。
- 核心链路、并发、性能、安全、权限、隐私、状态一致性、数据完整性相关任务。
- 系统重构、技术迁移、组件化、模块化、复用资产沉淀。
- 需要代码 review、复用评估、复用说明文档或工程交付记录的任务。

对于极小的局部改动，ECD 会走更轻量的流程，但仍会遵守项目风格、最小改动和验证原则。

## 工作流程

ECD 的常规工作流是：

1. 判断风险等级：Low、Medium、High、Prod-critical。
2. 检查项目结构、技术栈、入口、模块边界、代码风格、测试方式和运行方式。
3. 对 Medium 及以上任务先输出中文设计方案、影响范围、验收标准和确认点。
4. 用户确认后再修改代码、配置、脚本、文档或其他工程产物。
5. 修改时保持最小影响范围，遵循现有架构和命名风格。
6. 补充必要测试，或说明为什么不能自动化测试。
7. 执行可行验证，例如 lint、typecheck、unit test、build、集成检查、浏览器检查、dry-run、部署计划等。
8. 在 `docs/work/` 下生成中文任务交付与评审记录。

## 风险等级

- `Low`：局部修改，不影响公共契约、核心流程、状态、数据、部署或权限安全。
- `Medium`：新增行为、UI 流程、集成逻辑、状态处理、自动化或多文件调整。
- `High`：影响公共契约、存储、迁移、权限、安全、并发、性能、核心链路、部署或兼容性。
- `Prod-critical`：涉及生产稳定性、用户信任、资金、权限、隐私、合规、不可逆数据或关键业务流程。

风险越高，ECD 要求的方案设计、回滚策略、验证深度和文档记录越完整。

## Reference Modes

ECD 目前包含两个重要模式：`Review Mode` 和 `Reuse Mode`。

### Review Mode

当你说：

```text
ECD review
用 ECD review 一下
按 ECD 的 review 模式检查
review current changes
code review this PR
检查这次改动有没有问题
```

ECD 会进入代码评审模式。

Review Mode 的重点是先发现问题，而不是直接修复问题。它会检查：

- 正确性和行为变化。
- SOLID、职责边界和架构一致性。
- 安全、权限、隐私和敏感信息。
- 可靠性、事务、幂等、并发和数据一致性。
- 性能、资源使用、10 倍数据量或并发下的风险。
- 错误处理、日志、监控和可观测性。
- 测试覆盖、验证方式和残余风险。
- 可以删除或后续治理的代码。

评审输出会按 `P0` 到 `P3` 排序：

- `P0 Critical`：必须阻塞，可能导致安全漏洞、数据丢失、生产故障或严重正确性问题。
- `P1 High`：建议阻塞，可能导致逻辑错误、契约破坏、架构问题、权限或一致性风险。
- `P2 Medium`：通常建议修复或跟踪，包含边界条件、验证不足、维护性或测试缺口。
- `P3 Low`：低风险建议，例如命名、清晰度、小型清理。

除非用户明确要求修复，否则 Review Mode 不会在同一轮直接改代码。

### Reuse Mode

当你说：

```text
这个能复用吗
把这个做成可复用的
写复用说明文档
这个怎么迁移到新项目
打包成可复用组件/模块/工具/服务/脚本/模板
```

ECD 会进入复用模式。

Reuse Mode 适用于任何开发场景，不局限于前端或后端。可复用对象可以是：

- 前端组件、hooks/composables、样式、设计 token、页面级工具。
- 后端服务、领域模块、API client、middleware、job、adapter、repository。
- CLI、脚本、自动化流程、代码生成器、项目模板。
- 数据/ML pipeline、ETL、模型封装、数据校验器、报表生成器。
- 数据库 schema、migration、seed、存储过程、索引、查询模板。
- Dockerfile、Helm chart、Terraform/IaC 模块、CI/CD workflow、监控告警配置。
- 测试工具、fixtures、mocks、契约测试、E2E helper、QA checklist。
- 文档模板、运行手册、发布模板、知识库模板。

## 复用模式原则

ECD 对“复用”的定义很严格：

- 复用不是让用户复制散落的代码片段。
- 复用必须围绕文件、目录、模块、包、工具、服务、模板、制品或基础设施模块组织。
- 新项目必须能通过复制/安装完整复用单元，然后配置、导入/调用/执行/部署，并验证它能工作。
- 复用说明必须是可执行的，不能只写“未来可以抽离”“按业务调整”“建议统一封装”这类空泛建议。

Reuse Mode 会先检查目标是否真的具备复用性质：

- 输入是否通过 props、参数、配置、API 契约、CLI 参数、文件、schema、环境变量或适配器显式注入。
- 输出是否通过事件、返回值、文件、回调、API 响应、数据库状态、部署资源、日志或指标显式体现。
- 是否摆脱了当前页面、当前服务、当前数据库、当前队列、当前云账号、当前路由、全局状态、散落 CSS、硬编码 token 等耦合。
- 新项目是否只需要改配置，而不是修改内部源码。
- 是否有稳定入口，例如 `index.js`、包导出、CLI 入口、API module、migration 根目录、Docker image、Helm chart、Terraform module 或模板路径。
- 是否有验证路径证明它在新项目或隔离环境中可运行。

如果目标已经具备复用性质，ECD 会直接分析资产边界、添加必要资产标记，并输出复用说明文档。

如果目标不具备复用性质，ECD 会先说明问题和改造要求；在用户要求继续时进行可复用化改造，完成后必须验证，并先让用户确认改造结果。只有用户确认后，才会加入复用资产标记并生成复用文档。

## 复用资产标记

当文件被正式纳入 ECD 复用资产后，主要资产文件顶部必须有显式标记。常见形式如下：

```js
/**
 * @asset
 * @file MyCustomSlider.vue
 * @layer Component / UI Asset
 * @description 【ECD 核心技术资产】全栈通用的高性能滑动条组件。
 * @reusable true
 * @consumers [商城客户端, 个人中心, H5活动页]
 * @reuseDoc docs/reuse/[R]复用-高性能滑动条组件说明.md
 * @notice 本文件已作为 ECD 复用资产归档。修改本文件的行为、契约、配置、样式、验证方式或限制时，必须同步更新 @reuseDoc 指向的复用说明文档，以确保复用可靠性。
 */
```

未来 AI 或开发者读到 `@reusable true` 时，应当把这个文件视为高影响范围的共享资产：

- 先阅读资产头部信息。
- 查看 `@consumers` 了解已知使用方。
- 查看 `@reuseDoc` 指向的复用说明文档。
- 遵守 `@notice`，修改行为、契约、配置、验证方式或限制时同步更新复用文档。
- 不在未确认影响范围的情况下随意破坏兼容性。

如果文件格式不能安全写注释，需要通过相邻的 `.asset.md` 资产清单、包装入口、生成器或 manifest 承载同等信息。

## 文档输出规范

ECD 主要生成两类文档。

### 任务交付与评审记录

常规开发、修复、重构和配置变更完成后，ECD 会在项目下生成：

```text
docs/work/WyyyyMMdd（做了XXX）任务记录.md
```

文档内容会根据风险等级裁剪，至少记录：

- 任务背景和改动概述。
- 风险定级。
- 变更文件列表。
- 实现内容。
- 验收标准与验证结果。
- 当前设计的不足。

Medium 及以上任务还会记录项目现状、方案设计、影响范围和未来更好的设计；High/Prod-critical 任务还会补充风险说明、回滚方案和维护建议。

### 复用说明文档

复用资产文档统一放在：

```text
docs/reuse/[R]复用-XXXXX说明.md
```

复用文档默认使用中文，并包含：

- 文档生成时间。
- 复用对象。
- 来源项目/路径。
- 资产标记。
- 适用范围。
- 维护说明。
- `## 介绍`
- `## 实现原理`
- `## 详细原理`
- `## 分析`
- `### 可能的优势`
- `### 可能存在的缺陷`
- `## 复用方式`

其中 `## 复用方式` 必须写到新项目可以真正落地，包括前置条件、复制或安装方式、依赖安装、配置项、导入/调用/执行/部署路径、输入契约、输出处理、运行环境、权限/密钥、验证命令和预期结果。

## 使用示例

常规开发：

```text
使用 ECD，先看项目架构，再给我设计方案，确认后再编码。
```

代码评审：

```text
用 ECD review 模式检查当前改动。
```

复用评估：

```text
调用 ECD 复用模式，看看这个模块是否具备复用性质。
```

生成复用资产：

```text
调用 ECD 复用模式，把这个组件做成可复用资产，并输出复用说明文档。
```

维护复用资产：

```text
这个文件里有 @reusable true，修改前先看对应 reuse 文档，并同步更新说明。
```

## 目录结构

当前 ECD Skill 的关键文件如下：

```text
enterprise-code-design/
├── SKILL.md
├── README.md
├── agents/
│   └── openai.yaml
└── references/
    └── rules/
        ├── reuse-mode.md
        └── review-mode.md
```

- `SKILL.md`：核心触发说明、主流程、风险分级、代码规则、验证和交付规则。
- `references/rules/reuse-mode.md`：复用模式的详细规则。
- `references/rules/review-mode.md`：Review 模式的详细规则。
- `agents/openai.yaml`：Skill 在界面或列表中的展示元信息。

## 维护注意事项

- 修改 ECD 行为时，优先改 `SKILL.md` 或 `references/rules/*.md`，README 只同步人类可读说明。
- Skill 内部执行规则建议使用英文，生成给项目团队看的交付文档和复用文档默认使用中文。
- 新增模式时需要在 `SKILL.md` 中注册，并把详细规则放入 `references/rules/`。
- 修改 `Review Mode` 或 `Reuse Mode` 时，应同步检查 README 是否过期。
- 如果使用 `skill-creator` 的校验脚本失败，应至少手动检查 frontmatter、路径引用、触发描述和关键规则是否一致。
