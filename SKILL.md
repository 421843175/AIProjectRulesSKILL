# Enterprise Code Design

## 核心原则

当用户需要编写代码且任务比较复杂时，使用本技能。复杂任务包括但不限于：新增业务功能、跨模块改造、接口或数据库变更、核心链路改造、并发或性能问题、权限安全、重构、架构设计、技术栈选择、迁移任务、影响范围不清晰的需求。

简单、孤立、低风险的代码改动可以不触发完整设计确认流程，但仍应遵守项目风格和基础质量要求。

## 工作流

### 0. 任务准入与风险分级

复杂编码前先判断风险等级，并据此决定设计、测试、发布和文档深度：

- 低风险：局部小改，无接口、数据库、核心链路影响，可轻量说明后执行。
- 中风险：新增业务逻辑或改动多个文件，必须给设计方案、验收标准和测试方式。
- 高风险：涉及旧接口、数据库、并发、性能、安全、核心链路，必须确认方案、影响范围、回滚策略和专项检查。
- 生产关键风险：可能影响线上稳定性、数据一致性或用户资金/权限/隐私，必须先出编码计划流程文档，并明确灰度、监控、回滚和验收口径。

### 1. 先调研当前项目架构

在提出方案或编码前，必须先查看当前项目结构与既有实现方式。优先收集以下信息：

- 目录结构、模块边界、包划分、入口文件、构建和运行脚本。
- 当前使用的技术栈、框架、依赖、测试框架、日志库、配置方式。
- 现有架构风格：分层架构、DDD、MVC、微服务、插件化、事件驱动、仓储模式、工厂模式、策略模式、适配器模式等。
- 命名习惯：文件名、类名、方法名、接口名、DTO/VO/Entity/Service/Repository/Controller 等命名规律。
- 代码风格：异常处理、返回值结构、日志格式、参数校验、常量定义、配置读取、注释风格。
- 数据边界：数据库表、缓存、消息队列、第三方接口、旧接口兼容点。
- 测试现状：单元测试、集成测试、端到端测试、测试命名和目录习惯。

调研后，用简短中文总结项目现状，包括：架构类型、关键模块、设计模式、命名风格、潜在约束、与本需求相关的已有代码。

### 2. 编码前必须给出设计方案并等待确认

在复杂任务中，禁止直接进入编码。必须先给用户设计方案，并明确询问用户确认或修改。用户确认前不得修改业务代码。

设计方案必须包含：

- 需求理解：用中文复述目标、输入输出、边界条件和不确定点。
- 项目现状：基于调研结果说明当前架构、相关模块、已有风格。
- 技术栈选择：说明选择现有技术栈还是引入新技术，并解释理由。优先沿用项目已有技术栈，只有收益明确时才引入新依赖。
- 依赖治理：新增依赖必须说明必要性、替代方案、维护状态、License、安全风险、体积或运行成本。
- 企业级架构思考：说明分层、模块或包划分、职责边界、数据流、接口契约、事务边界、错误处理、扩展点、兼容策略。
- 设计规范：模块、包、类、方法、接口、DTO、配置、常量、测试目录必须符合企业设计规范，并贴合当前项目已有习惯。
- 方案内容：可以给出一个方案，也可以给出多个方案。
- 推荐方案：如果存在多个方案，必须给出推荐项和推荐理由。
- 验收标准：明确什么算完成，包括正常路径、异常路径、边界值、性能、兼容性和用户可观察结果。
- 用户确认点：明确列出需要用户确认或选择的内容。

### 3. 多方案时必须提供评估维度

如果给出多种设计方案，必须提供客观对比，至少包含以下维度：

- 时间/复杂度成本：例如方案 A 快速但可能产生技术债，方案 B 更优雅但改动更大。
- 可扩展性与性能：说明高并发、大数据量、长期迭代下的表现。
- 影响范围（Blast Radius）：显式列出会影响哪些现有模块、旧接口、数据库表、缓存、配置、任务、消息、前端页面、测试或部署流程。

即使只有一个方案，也应说明该方案的时间成本、扩展性、性能风险和影响范围。

### 4. 大厂级专项检查（按需）

高风险或生产关键风险任务必须补充专项检查，内容保持简短但结论明确：

- 发布与回滚：说明是否需要灰度、Feature Flag、兼容旧版本、数据迁移、回滚步骤和上线后观察点。
- 安全与隐私：检查权限、输入校验、注入风险、密钥泄漏、敏感信息脱敏，日志禁止输出密码、Token、密钥等敏感数据。
- 数据变更：涉及数据库时说明 migration、索引、历史数据、事务、幂等、回滚和数据一致性验证。
- 可观测性：说明关键日志字段、指标、Trace、错误率、耗时、吞吐量、告警条件；项目无指标系统时至少保证日志可定位。
- 性能基线：涉及高并发、大数据量、实时处理或批任务时，说明当前基线、目标指标、瓶颈和压测/验证方式。
- 技术债记录：若采用快速或临时方案，必须记录技术债、原因、影响和后续治理建议。

### 5. 非常复杂时先出具编码计划流程文档

当任务非常复杂、跨多个阶段、涉及数据库迁移或多系统协作时，编码前必须先出具编码计划流程文档。文档可以直接写在回复中，或在用户同意后落地为项目内文档。

编码计划流程文档必须包含：

- 阶段拆分：调研、设计、接口/数据模型、实现、迁移、测试、灰度或回滚。
- 每阶段目标：说明每一步完成什么。
- 文件级计划：列出预计新增或修改的文件。
- 风险点：兼容性、数据一致性、性能、并发、安全、回滚、测试缺口。
- 验证方式：每阶段如何验证。
- 执行顺序：按依赖关系一步步执行。

用户确认编码计划后，再按计划逐步执行，并在关键节点同步进展。

## 编码硬性要求

确认方案后再编码。编码必须遵守以下要求。

### 变更隔离与最小影响

- 修改代码必须保持最小影响范围，只处理与当前需求直接相关的代码、配置、测试和文档。
- 严禁改动与当前任务无关的代码、注释、格式、命名、依赖、配置或文档。
- 不得为了顺手优化、统一风格、重排格式、清理注释而触碰无关文件或无关代码块。
- 必须避免影响既有功能；如存在影响风险，设计阶段必须显式说明影响范围、兼容策略、验证方式和回滚方案。
- 如发现无关代码存在问题，只能在回复或复盘文档中记录建议，除非用户明确授权，否则不得修改。
### 中文注释

- 新增或修改的每个代码行上方都要添加中文行注释，解释该行目的。
- 方法上方必须添加中文块注释，描述方法作用、参数含义、返回值、异常或副作用。
- 类或文件开头必须添加中文块注释，描述该类或文件的职责、所属层级、使用场景和注意事项。
- 注释必须解释业务意图和设计原因，避免无意义复述语法。
- 如果项目已有更严格或冲突的注释规范，先向用户说明冲突，并请求确认后执行。

### 工程健壮性

- 防御性编程：必须进行严格参数校验，包括非空、类型、边界值、枚举值、权限或状态前置条件。
- 异常处理：核心步骤必须包含异常捕获、错误分类、上下文日志和优雅降级。不要吞异常。
- 可观测性：关键业务路径必须打日志，日志必须携带上下文特征，例如 orderId、userId、requestId、taskId、traceId、entityId。
- 严禁 Hard Code：魔术数字、硬编码字符串、超时时间、开关、路径、状态码、业务常量必须抽取为常量、枚举或配置文件。
- 数据一致性：涉及写操作时说明事务边界、幂等策略、并发控制、重试和回滚行为。
- 安全性：涉及外部输入、文件、网络、权限、SQL、命令执行时必须校验、转义或使用安全 API。

### 企业级代码组织

- 模块或包划分必须遵循企业设计规范，并贴合当前项目已有架构。
- 按职责拆分 Controller/API、Application/Service、Domain、Repository/DAO、Infrastructure、DTO、Config、Constants、Tests 等层级；具体命名以项目现状为准。
- 不把业务逻辑堆在入口、Controller、脚本或 UI 层。
- 不引入无必要的大型抽象；抽象必须服务于复用、扩展或降低复杂度。
- 新接口要考虑向后兼容、版本化、错误码、返回结构和调用方迁移。

## 测试与交付

实现完成后必须编写测试用例，或在无法直接编写测试时明确告知用户测试方式和原因。

优先级如下：

1. 能写自动化测试时，必须补充与项目测试框架一致的测试用例。
2. 涉及核心业务逻辑时，至少覆盖正常路径、参数非法、边界值、异常路径、幂等或并发风险点。
3. 涉及接口或数据库时，优先补充集成测试或给出可执行的接口/SQL 验证步骤。
4. 涉及前端时，补充组件测试、交互测试，或使用浏览器验证关键流程。
5. 如果因为环境、依赖、外部服务、权限或时间限制无法编写测试，必须在最终回复中说明原因，并给出用户可执行的测试命令、测试数据、验证步骤和预期结果。

交付前必须执行可行的验证命令，例如 lint、typecheck、unit test、integration test、build 或项目已有测试脚本。最终回复中必须说明执行了哪些验证、结果如何、还有哪些剩余风险。

### 完成定义（Definition of Done）

最终交付必须满足 DoD：方案已确认、代码已实现、测试或测试方式已补充、验证命令已执行、关键日志/错误处理/配置抽取已覆盖、发布回滚风险已说明、Markdown 复盘文档已生成或已说明未生成原因。

### 测试完成后的 Markdown 复盘文档

测试完成后，必须出具一份 Markdown 文档，记录本次复杂编码任务的完整过程。优先放在项目已有文档目录中，例如 `docs/`、`doc/`、`documentation/`、`设计文档/`；如果项目没有文档目录，可在用户确认后新建合适目录，或在最终回复中直接提供 Markdown 内容并说明未落地成文件的原因。

Markdown 文档文件名必须便于日后按时间追溯发生过什么，采用 `yyyyMMdd（做了XXX）文档.md` 或 `yyyyMMdd（做了XXX）任务记录.md` 格式。日期使用任务完成或测试完成当天的 8 位年月日，括号内用简短中文概括本次核心工作，避免泛泛命名。示例：`20260609新增RTCM解码任务记录.md`、`20260609（新增RTCM解码）任务记录.md`、`20260609（修复订单回调幂等）文档.md`。

Markdown 文档必须包含：

- 任务背景：用户原始目标、关键约束、复杂度判断依据。
- 讨论核心内容：与用户确认过的关键问题、用户选择、需求变更、取舍结论。
- 项目架构调研：当前项目架构、模块划分、设计模式、命名风格、技术栈、相关旧接口或数据库表。
- 设计方案：最终采用的方案、备选方案、推荐理由、评估维度、验收标准、影响范围（Blast Radius）。
- 实现流程：按时间或阶段说明做了哪些修改、涉及哪些文件、核心代码职责、相关配置或提交信息。
- 测试方式：写了哪些测试用例、覆盖了哪些场景、执行了哪些命令、测试数据是什么、预期结果和实际结果。
- 怎么看结果：告诉用户如何运行、如何查看页面/API/日志/数据库/测试报告，以及如何判断功能成功。
- 分析足与不足：总结本次实现做得好的点、仍然不足的点、潜在风险、技术债、后续优化建议和维护边界。
- 回滚与排查：如果上线或使用后出问题，说明如何回滚、如何定位日志、如何复现和排查。

最终回复中必须给出该 Markdown 文档的文件路径；如果没有生成文件，必须说明原因，并把完整 Markdown 文档内容直接提供给用户。

## 回复格式要求

设计阶段回复建议使用以下结构：

```markdown
**项目现状**
...

**方案设计**
...

**评估维度**
...

**影响范围**
...

**推荐方案**
...

**请确认**
...
```

实现完成后的最终回复建议包含：

- 修改了什么。
- 涉及哪些关键文件。
- 已编写哪些测试或为什么无法编写测试。
- 已执行哪些验证命令及结果。
- Markdown 复盘文档路径，或未生成文件时的完整 Markdown 文档内容。
- 剩余风险或需要用户确认的事项。
